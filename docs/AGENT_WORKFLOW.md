## 0. Ground Truth: what this repo actually is

Before any agent touches code, internalize the coupling that dictates what may run in parallel.

**The compile pipeline is a single linear data flow through tightly shared types:**

```
src/parser.rs   parse(&str) -> ast::Stylesheet
src/eval.rs     Evaluator::eval_sheet(&Stylesheet) -> Vec<OutNode>
src/emit.rs     emit(&[OutNode], OutputStyle) -> String
                          (lib.rs::compile ties the three together)
```

- **`src/ast.rs`** is the contract every other module imports: `Stmt` (today only `VarDecl | Rule | Decl | Import | Comment` — *no control-flow, mixin, or function nodes*), `Expr` (`Number | Color | QuotedString | Ident | Bool | Null | Var | Binary | Unary | Func | List | Paren | Interp`), `BinOp` (`Add | Sub | Mul | Mod` — *no `Div`, no comparison, no `and/or/not`*), `UnOp` (`Neg` only). **Every new language feature adds a variant here, and a new variant forces a new match arm in `parser.rs`, `eval.rs`, and often `emit.rs` simultaneously.** This is the central serialization point.
- **`src/value.rs`** defines `Value` (`Number | Color | Str | List | Bool | Null` — *no Map, no function reference, no `arglist`*). `Value::to_css`, `to_interp`, `type_name`, and `fmt_num` are exhaustive matches; adding a `Value` variant breaks every one of them at compile time.
- **`src/eval.rs::Evaluator`** holds `scopes: Vec<HashMap<String,Value>>`, `eval_expr`, `eval_binary`, and the statement dispatch in `eval_top_stmts`/`eval_rule`. Control flow (`@if/@for/@each/@while`), `@mixin/@include`, and `@function` all rewrite *this* dispatch — they are inherently sequential against each other.
- **`src/builtins.rs::call`** is a flat `match name { ... }` with a `_ => plain_css_function(...)` catch-all. **This is the one place where parallel work is nearly conflict-free**: each builtin is an independent arm + an independent `fn fn_*`.
- **`src/parser.rs::parse_at_rule`** today handles only `@import` and returns `Err("@{other} is not supported in this build")` for everything else. Every new at-rule is a new arm here.

**The conformance harness `spec/run_spec.py`** is the source of truth for the ratchet. Key facts an agent must know:
- Flags: `--filter <substr>` (case-name substring), `--limit N`, `--style {expanded,compressed}`, `--impl <name>`, `--out <path>`, `--quiet`, `--no-skip <tag>`, `--run-skipped`.
- It scores against the **dart-sass** expectation (`output-dart-sass.css` overrides `output.css`; `error-dart-sass` overrides `error`).
- `SASS_BIN` (default `target/release/sasso`) names the binary under test; relative paths resolve against the repo root.
- It writes `results.json` with `summary` (`total`, `attempted`, `pass`, `error_expected`, `fail`, `skip`, `pass_including_error_expected`, `pass_pct_of_attempted`, `skip_breakdown`) and a `cases` array of `{name, status, skip_tag, reason, input_name}` where `status ∈ {PASS, FAIL, ERROR_EXPECTED, SKIP}`.
- **Skip tags** (`indented-syntax`, `use`, `forward`, `extend`) carve out features that are out of scope *today*. The ratchet measures `attempted`, not `total` — so retiring a skip tag (e.g. implementing `@extend`) **expands the denominator** and must be a deliberate, baseline-bumping event, never an accident.
- The CLI it drives accepts `--style=...` and a positional input path (see `src/main.rs`), and `-I`/`--load-path`. The harness writes `input.scss` + sibling `@import` files to a temp dir, so importer behavior is exercised through `FsImporter`.

**The two test layers that lock in coverage:**
- `tests/integration.rs` — deterministic goldens, every expected string produced by dart-sass 1.101 and verified byte-for-byte. **Runs offline on every `cargo test`. This is where regression protection lives.**
- `tests/parity.rs` — opt-in live diff vs dart-sass, gated by `SASSO_PARITY=1` + `SASS_BIN`. Returns early otherwise. CI runs it in a dedicated job.

**CI (`.github/workflows/ci.yml`)** has three jobs: `test` (`cargo fmt --check`, `cargo clippy --all-targets --all-features -D warnings`, `cargo test --all-features`), `parity` (installs dart-sass, runs `tests/parity.rs`), `wasm` (builds the lib for `wasm32-wasip1`). **`RUSTFLAGS: -D warnings` is global — a single clippy lint fails the build.** The ratchet adds a fourth gate (§4).

## 1. What Parallelizes Safely vs. What Must Be Sequential

The decisive question for every unit of work: **does it add or change a variant of a shared enum (`Stmt`, `Expr`, `Value`, `BinOp`, `UnOp`), or rewrite the `Evaluator` statement dispatch?** If yes → sequential. If it only adds a leaf (a builtin arm, a self-contained `fn`, a test, a read-only analysis) → parallel.

### 1a. SAFELY PARALLEL (fan these out to N agents at once)

1. **Read-only failure bucketing.** Agents read `results.json` and the failing cases' `input.scss`/expected output; they *write no source*. Produce a classification of every `FAIL` into a feature bucket (see §6 buckets). Pure analysis → unlimited parallelism. Multiple agents can bucket disjoint name-prefixes (`--filter spec/values`, `--filter spec/css`, …) concurrently.
2. **Adversarial verification of a candidate fix.** Given a branch/worktree that claims "+K passes," an independent verifier agent re-runs the harness, diffs `results.json`, and re-checks no previously-passing case regressed. Verifier agents share nothing mutable with the implementer — they operate on a built binary and JSON. Run several verifiers on the same candidate for high-value features.
3. **Independent test-fixture authoring.** Writing new `tests/integration.rs` goldens or `tests/parity.rs` cases for a *named* feature is append-only to a test file. Several agents can each draft fixtures for different features in scratch files; integration concatenates them (conflicts are trivial, append-only, never semantic).
4. **New builtin functions, one per agent.** Because `builtins::call` dispatches by name to an isolated `fn`, two agents implementing, say, `string.length`/`str-length` and `list.nth`/`nth` touch **disjoint match arms and disjoint functions**. The only shared edit is inserting a `name => ...` arm into the `call` match — a one-line, near-always-clean merge. *Caveat:* a builtin that needs a **new `Value` variant** (e.g. `map-get` needs `Value::Map`) is NOT in this class — it becomes sequential value-model work (§1b).
5. **Documentation / `spec/REPORT.md` regeneration, benchmark runs** (`bench/`) — read-only against `src/`.

### 1b. MUST BE SEQUENTIAL (one agent at a time, serialized through integration §2)

These all edit the same few hot files and/or the same enums. Two agents here *will* textually conflict on the match arms and *will* semantically conflict on evaluation order:

1. **Any new `Stmt` variant** — control flow (`@if/@else`, `@each`, `@for`, `@while`), `@mixin`+`@include`, `@function`+`@return`, `@media`/`@supports`, `@at-root`, nested `@import`. Each rewrites `parse_at_rule` (parser.rs:265), the `Stmt` enum (ast.rs:15), and the dispatch loops in `eval_top_stmts`/`eval_rule` (eval.rs:115/135). Control flow specifically rewrites the **statement evaluator's structure** (it must thread `out`/`items`/`nested` through loops and conditionals) — the single most conflict-prone change in the codebase. **At most one control-flow feature in flight at a time.**
2. **Any new `Expr` variant** — `Div`/comparison/`and`/`or`/`not` operators, maps `(k: v)`, function-reference `get-function`, `if()` ternary. Touches `Expr` (ast.rs:56), the Pratt/precedence parsing in `parser.rs::parse_value`, and `eval_expr`/`eval_binary` (eval.rs:228/301).
3. **Any new `Value` variant** — `Map`, `arglist`, function references, calculations. Breaks the exhaustive matches in `to_css`/`to_interp`/`type_name`/`fmt_num` and every consumer in `eval.rs`/`builtins.rs`. Highest blast radius; schedule alone.
4. **`BinOp`/`UnOp` extensions** — e.g. real `/` division semantics, `not`. Touches ast.rs, parser precedence, and `eval_binary`.
5. **Selector-engine rewrites** — `resolve_selectors`/`normalize_selector` (eval.rs:411+) for `@extend`, `:is()`, complex combinator edge cases. One owner; this code has subtle ordering invariants (parent×child product, bubble order).

**Rule of thumb for scheduling:** group in-flight work so that no two parallel worktrees both modify `ast.rs`, both modify `value.rs`, or both rewrite the `eval.rs` statement dispatch. Builtins-only worktrees are exempt and may run alongside one sequential feature.

## 2. Git-Worktree Isolation + Sequential Integration

**Goal:** implement independent features physically in parallel, but *land* them one at a time so the harness re-measurement catches semantic conflicts the textual merge cannot.

### 2a. Pick disjoint features per round (conflict minimization)

For a round of K parallel worktrees, choose features whose edits touch **disjoint match arms / functions**:

- **Best case (K can be large):** K distinct builtins. Each is `builtins.rs` arm + `fn fn_x`. The only shared line is the `call` match; cherry-picks rarely collide and never semantically conflict.
- **Mixed (K = 1 sequential + M builtins):** one `Stmt`/`Expr` feature *plus* several builtins. The control-flow worktree owns `ast.rs`/`eval.rs` dispatch; the builtins worktrees own `builtins.rs`. Disjoint files ⇒ clean cherry-picks.
- **Forbidden in one round:** two worktrees that each add a `Stmt` variant, or two that each add a `Value` variant. They share the same enum and the same `eval.rs` arms — guaranteed conflict, and worse, *silent* semantic interaction (e.g. both assume they run before the other in `eval_top_stmts`).

### 2b. The worktree lifecycle

```bash
# From repo root, one worktree per parallel feature, each off current master:
git worktree add ../sasso-feat-str-length  master
git worktree add ../sasso-feat-list-nth     master
git worktree add ../sasso-feat-if-else      master   # the lone sequential one
```

Each agent works entirely inside its worktree: edits, `cargo build --release`, `cargo test`, builds its own binary, and **measures its own delta against a frozen baseline** before declaring done:

```bash
# inside ../sasso-feat-str-length
cargo build --release
SASS_BIN=target/release/sasso python3 spec/run_spec.py \
  --filter spec/core_functions/string --out spec/results-feat.json --quiet
```

The agent commits one atomic commit (§3) on a feature branch *inside its worktree*. It does NOT merge.

### 2c. Sequential integration with re-measurement after each

The integrator (a dedicated agent or the orchestrator) lands worktrees **one at a time**, rebuilding and re-running the **full** harness after each — because a clean three-way merge can still produce a semantic regression (e.g. a new builtin shadows a name another feature relied on; two features both changed `fmt_num` rounding).

```bash
# On master, integrate in ROI order, lowest-risk (builtins) first:
git cherry-pick <str-length-commit>      # or: git rebase the feature branch
cargo build --release
SASS_BIN=target/release/sasso python3 spec/run_spec.py --out spec/results.json --quiet
python3 spec/ratchet.py check            # §4: pass-count must be >= baseline
python3 spec/ratchet.py diff spec/results.prev.json spec/results.json   # §5: no regressions
# Only if both green:
cp spec/results.json spec/results.prev.json
python3 spec/ratchet.py bump             # raise baseline to the new pass count
git add spec/BASELINE.json spec/results.prev.json && git commit --amend --no-edit
# then the next worktree:
git cherry-pick <list-nth-commit>
... (re-measure full suite again) ...
```

If integrating worktree *i* drops any previously-green case, **stop**: the merge exposed a semantic conflict. Resolve in a fresh fixup commit (and add a golden in `tests/integration.rs` capturing the interaction) before proceeding to worktree *i+1*. Re-running the full suite (not just `--filter`) after *every* integration is what makes this safe — a feature-scoped filter would miss a cross-feature regression.

After all worktrees land: `git worktree remove ../sasso-feat-*`.

## 3. Every Change Is One Atomic Commit

A landed change is a single commit that is self-contained and independently revertible. It MUST carry, together:

1. **The feature** — the `src/` edits (new builtin arm + fn, or new `Stmt`/`Expr`/`Value` variant + its parser/eval/emit arms).
2. **A locked-in regression test** — at least one golden in `tests/integration.rs` whose expected string was produced by and verified against dart-sass (the file's existing convention), **and/or** a `tests/parity.rs` case for the feature. The golden is mandatory because it runs offline on every `cargo test` and is the durable guard; the parity case is the live cross-check.
3. **A recorded pass-count delta** — the commit bumps `spec/BASELINE.json` (§4) and the commit message states the measured delta.

**Commit message shape** (conventional-commits; per the repo's existing `git log` style, e.g. `feat:`, `test:`, `ci:`):

```
feat(builtins): add str-length / string.length

Implements str-length($string) returning the character count as a
unitless number; unknown-function passthrough no longer swallows it.

spec: attempted 318 -> 318, pass 142 -> 149 (+7), no regressions
baseline: 142 -> 149
```

> Per this machine's global instruction, do **not** append any `Co-Authored-By` trailer to commits.

**Atomicity discipline:** never let a commit contain a feature without its test, or a baseline bump without the feature that earned it. If a fix turns out to regress a case, the *whole commit* reverts cleanly, restoring the baseline in lockstep. One feature = one commit; squash exploratory work in the worktree before integration.

## 4. The Committed Pass-Count BASELINE Ratchet

Mirror the rubyrs coverage-ratchet pattern: a committed floor that only ever rises, enforced in CI.

### 4a. `spec/BASELINE.json`

Commit a baseline keyed so a partial measurement can't be mistaken for the whole:

```json
{
  "suite_version": "<contents of spec/SPEC_VERSION.txt>",
  "impl": "dart-sass",
  "style": "expanded",
  "enabled_skip_tags": ["indented-syntax", "use", "forward", "extend"],
  "attempted": 318,
  "pass_including_error_expected": 149,
  "updated": "2026-06-06"
}
```

Why these fields:
- `pass_including_error_expected` is the ratcheted number — it matches the harness's own `summary.pass_including_error_expected` (PASS + ERROR_EXPECTED), so error-spec wins count.
- `attempted` is recorded so a **drop in attempted** (someone narrowed scope or a case vanished) is flagged, and a **rise in attempted from retiring a skip tag** is a conscious decision, not silent denominator inflation.
- `enabled_skip_tags` pins the scope. Changing this set (e.g. dropping `extend` once `@extend` works) is itself a baseline event reviewed on its own.
- `suite_version` pins to `spec/SPEC_VERSION.txt` so a suite bump and a code change can't be conflated.

### 4b. `spec/ratchet.py` (small companion to `run_spec.py`)

A thin script with three verbs the workflow already invokes in §2c:

- `check` — run/ load the full `results.json`, compare `summary.pass_including_error_expected` against `BASELINE.json`. **Fail (exit 1) if it regressed below baseline**, or if `attempted` dropped, or if `enabled_skip_tags`/`suite_version` drifted unexpectedly.
- `diff <prev.json> <cur.json>` — set-difference the per-case `cases[]` by `name`: any case that was `PASS`/`ERROR_EXPECTED` in `prev` and is now anything else is a **regression** → exit 1, printing the regressed case names. Newly-green cases are printed as the win.
- `bump` — rewrite `BASELINE.json` to the current `attempted` + `pass_including_error_expected`, refresh `updated`. Only run after `check` and `diff` pass.

### 4c. CI gate (new fourth job in `.github/workflows/ci.yml`)

```yaml
  ratchet:
    name: sass-spec ratchet
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4            # spec/sass-spec must be vendored or fetched
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - run: cargo build --release
      - run: SASS_BIN=target/release/sasso python3 spec/run_spec.py --out spec/results.ci.json --quiet
      - run: python3 spec/ratchet.py check            # fails if pass count < BASELINE
      - run: python3 spec/ratchet.py diff spec/results.prev.json spec/results.ci.json
```

Note: `run_spec.py` already `sys.exit(1 if counts["FAIL"] else 0)` — so it returns non-zero while *any* case still fails. The ratchet job therefore relies on `ratchet.py check`/`diff` for its pass/fail decision (they exit 0 when the floor holds), not on `run_spec.py`'s own exit code. A PR is mergeable iff the ratchet did not regress; every feature PR raises the floor.

## 5. The Adversarial Verification Loop Before a Commit Lands

No commit lands on `master` until it has *proven*, against the full suite, that it strictly helped and harmed nothing. This runs at two moments: inside the worktree (self-check) and again at integration (independent re-check).

**Step 1 — Freeze the prior full-suite result.** Before the round, on clean `master`: `cargo build --release && SASS_BIN=target/release/sasso python3 spec/run_spec.py --out spec/results.prev.json --quiet`. Commit `results.prev.json` so the diff baseline is reproducible.

**Step 2 — Build the candidate and re-run the FULL harness.** Not a filtered subset — `--filter` is for *fast inner-loop iteration* while implementing, but the landing gate is the whole suite, because cross-feature regressions hide outside the filter.

**Step 3 — Prove strict improvement.** `summary.pass_including_error_expected` (candidate) must be **strictly greater** than the prior run's. A feature that adds a builtin but flips zero cases green is not landable as a conformance win (it may still land as scaffolding, but then it carries no baseline bump and must be justified).

**Step 4 — Prove zero regression.** `python3 spec/ratchet.py diff spec/results.prev.json spec/results.json`. Any `name` that was `PASS`/`ERROR_EXPECTED` and is no longer → the commit is rejected. This is the load-bearing check: it's easy to fix `spec/values/numbers` while breaking `spec/values/colors`.

**Step 5 — Parity gate.** Run the live cross-check for the feature: `SASSO_PARITY=1 SASS_BIN=sass cargo test --test parity`. The new feature's golden in `tests/integration.rs` must already match dart-sass byte-for-byte (that's how the goldens are authored). The parity test confirms it dynamically, catching cases where the hand-copied golden was itself wrong.

**Step 6 — Independent verifier agents re-check the diff.** A *separate* agent (sharing no mutable state with the implementer) re-runs Steps 2–5 from the cherry-picked commit on a clean tree and independently confirms the regression diff is empty and the win is real. For high-blast-radius changes (`Value`/`eval.rs` dispatch), run **two** verifiers; they must agree. The verifier also runs `cargo fmt --check` and `cargo clippy --all-targets -D warnings` because CI's global `RUSTFLAGS: -D warnings` will reject a lint regardless of conformance.

Only when Steps 3, 4, 5, and 6 all pass does the integrator run `ratchet.py bump`, refresh `results.prev.json`, and let the atomic commit land.

## 6. The Fan-Out / Fan-In Loop (loop-until-dry)

Run rounds until `summary.fail == 0` across the whole suite at the current scope, then retire a skip tag and continue.

**Round structure:**

**(A) Fan-out: bucket failures (read-only, parallel — §1a).**
Spawn bucketing agents over disjoint name-prefixes (`--filter spec/values`, `spec/css`, `spec/core_functions`, `spec/directives`, `spec/expressions`, …; the repo already has per-area `results-analyze-*.json` snapshots to seed this). Each classifies its `FAIL` cases into feature buckets. Expected dominant buckets, given §0's gaps:
- **Control flow** (`@if/@else`, `@for`, `@each`, `@while`) — sequential, one at a time.
- **Mixins/functions** (`@mixin`/`@include`, `@function`/`@return`) — sequential.
- **Operators** (`/` division, `==`/`!=`/`<`/`>`, `and`/`or`/`not`, `==` on lists) — sequential (`Expr`/`BinOp`).
- **Maps** (`(k: v)`, `map-get`, …) — sequential (`Value::Map`).
- **Builtins** (string, list, math, meta, color functions not yet present) — **parallel**, the ROI workhorse.
- **Emit/serialization** edge cases (number formatting, compressed loud comments, selector normalization) — small, often sequential but low-conflict.

**(B) Score ROI per bucket.** ROI ≈ (cases the feature would flip green) ÷ (implementation blast radius). Builtins typically win early rounds: each flips several `spec/core_functions/*` cases at near-zero conflict cost. Control flow has huge case counts but high blast radius — schedule exactly one per round.

**(C) Fan-out: implement top-ROI *independent* features in parallel worktrees (§2a/§2b).** Compose each round as **≤1 sequential feature + as many disjoint builtins as agents allow.** Verify each feature must touch disjoint match arms before dispatching (no two worktrees own `ast.rs`, none both own `value.rs`).

**(D) Adversarial verification per worktree (§5), in parallel.** Each implementer's claim is independently re-checked by a verifier agent against the frozen `results.prev.json`.

**(E) Fan-in: integrate sequentially (§2c).** Cherry-pick lowest-risk (builtins) first, then the one sequential feature. **Rebuild and re-run the FULL suite after every single integration.** Resolve any merge-exposed semantic regression with a fixup commit + an `integration.rs` golden capturing the interaction.

**(F) Re-measure and bump (§4/§5).** After the round fully integrates, the new full-suite pass count is the round's result. `ratchet.py bump` raises `BASELINE.json`; `results.prev.json` is refreshed for the next round.

**(G) Loop.** Re-bucket the now-smaller `FAIL` set and start the next round. When a whole scope is exhausted (`fail == 0`), **retire a skip tag deliberately**: drop e.g. `"extend"` from `BASELINE.json`'s `enabled_skip_tags` and from the harness invocation. This expands `attempted`, surfaces a fresh failure population (the `@extend`/placeholder cases), and the loop resumes — now growing the *denominator* toward the true 100% of the full sass-spec suite.

**Termination:** all skip tags retired AND `summary.fail == 0` over the entire `spec/sass-spec/spec` suite at the pinned `suite_version` ⇒ 100% conformance. The ratchet guarantees that from the first round to the last, the committed floor never falls and every merged commit is a strict, independently-verified, regression-free step up.

## 7. Quick-Reference Command Crib (for every agent)

```bash
# Build the binary the harness drives
cargo build --release

# Fast inner loop while implementing one feature (filtered, offline)
SASS_BIN=target/release/sasso python3 spec/run_spec.py \
  --filter spec/core_functions/string --out spec/results-feat.json --quiet

# Full-suite landing measurement (the gate)
SASS_BIN=target/release/sasso python3 spec/run_spec.py --out spec/results.json --quiet

# Regression diff + ratchet (companion script, §4)
python3 spec/ratchet.py diff spec/results.prev.json spec/results.json
python3 spec/ratchet.py check          # exit 1 if pass count < BASELINE
python3 spec/ratchet.py bump           # only after check + diff pass

# Offline regression goldens (must pass on every commit)
cargo test --test integration

# Live parity (the cross-check; needs dart-sass on PATH)
SASSO_PARITY=1 SASS_BIN=sass cargo test --test parity

# What CI enforces globally — never skip these before landing
cargo fmt --all --check
cargo clippy --all-targets --all-features -- -D warnings

# Worktree isolation
git worktree add ../sasso-feat-NAME master
git worktree remove ../sasso-feat-NAME
```

**The one invariant that makes the whole system safe:** after *every* integration, rebuild and re-run the **full** spec harness, diff per-case against the prior full run, and refuse to bump the baseline unless the pass count strictly rose and the regression set is empty.

## Real-run note: builtin fan-out (2026-06-06)

The first parallel builtin phase (math, string, list, meta, color_ext)
was attempted with `Workflow` `isolation: 'worktree'` but failed:
`Cannot create agent worktree: not in a git repository`. Root cause: the
session began **before** `git init`, and the worktree feature keys off the
session-start git detection, so a mid-session repo is invisible to it.

Fallback used: the same workflow run **sequentially** (a `for…await` loop,
no isolation) in the shared checkout — each agent built, verified against
dart-sass, and committed its own disjoint module file before the next ran,
so there was no race on shared build artifacts. Result: five families,
each byte-verified and atomically committed, sass-spec **1267 → 1351
(+84)**, no regression.

Takeaway for the next large parallel phase: start a **fresh conversation**
in the now-existing git repo so worktree isolation (true parallelism) is
available. Caveat: expect ~2–3× wall-clock (not 5×) — N concurrent
`cargo build` + spec runs + `npx sass` contend for one machine — so reserve
it for big batches. The disjoint per-family module layout makes both modes
clean (parallel integrates by cherry-pick; serial commits straight to
master).
