# dart-sass vs grass — SCSS compiler benchmark

First run of the `bench/` harness. A third engine (`sasso`) will be slotted
in later; see [README.md](./README.md) for how.

## Machine & versions

| Item | Value |
|---|---|
| CPU | Apple M2 Max (8 performance + 4 efficiency cores) |
| `hw.ncpu` | 12 |
| OS | macOS, Darwin 25.3.0 |
| Rust | `cargo 1.95.0 (f2d3ce0bd 2026-03-21)` (Homebrew) |
| dart-sass | `1.100.0 compiled with dart2js 3.12.0` (via `npx sass`) |
| grass | `0.13.4`, `default-features = false` |
| hyperfine | present (`/opt/homebrew/bin/hyperfine`) |

> Note on the dart-sass build: `npx sass` runs the **dart2js / Node.js** build of
> dart-sass, *not* the AOT-compiled native (`sass` Homebrew/`.exe`) build. Its
> startup is therefore Node+JS startup (~137 ms), and its compute runs on the V8
> JIT. A native AOT dart-sass would have lower startup (~tens of ms) and somewhat
> faster compute. This is called out because the task brief assumed AOT-native;
> what's actually on this machine is dart2js. All dart-sass numbers below are the
> dart2js build.

## Corpus

| File | What it is | dart-sass output |
|---|---|---|
| `corpus/handwritten/main.scss` (+ `partials/`) | Hand-written: variables, maps, nesting, `&`, `@mixin`/`@include` (incl. `@content`), `@each`, `@for`, `lighten`/`darken`/`mix`/`rgba`, `#{}` interpolation, `@import` partials | ~325 lines |
| `corpus/generated/large.scss` | Generated by `scripts/gen_corpus.rb` (400 components × variants × sizes via `@each`/`@for` + a color-math mixin) | ~25,999 lines |
| `corpus/batch/*.scss` (40 files) | 40 generated 60-component files, for amortized-startup measurement | ~25 k lines total |

The generated `large.scss` takes the dart2js dart-sass build ~360 ms cold and
~105 ms pure-compute, i.e. a meaningful fraction of a second — non-trivial
per-compile work, as required.

## Results

All times are hyperfine mean ± σ. "Pure compile" = process startup removed.

| Engine | Startup (ms) | Cold single-file, large (ms) | Pure compile, large (ms/file) | Pure throughput (files/sec) |
|---|---|---|---|---|
| **grass 0.13.4** (in-process) | **1.4 ± 0.3** | **27.4 ± 0.8** | **21.4** | **46.8** |
| **dart-sass 1.100** (direct bin) | 137.3 ± 3.3 | 359.0 ± 2.8 | 105.4 | 9.5 |
| **dart-sass via `npx sass`** | 1079 ± 228 | 1606 ± 142 | (same compute as above) | — |

Amortized-batch cross-check (40 medium files, **one** invocation each):

| Engine | Batch wall (ms) | Per-file (ms) |
|---|---|---|
| grass (40 files, in-process, sequential) | 137.7 ± 1.6 | 3.4 |
| dart-sass (dir→dir, **parallel isolates**) | 933.9 ± 7.0 | 23.3 |

### Headline numbers

- **Startup cost is the big subprocess story.** grass in-process pays ~**1.4 ms**;
  the dart-sass bin pays ~**137 ms** (≈ **98× more**); `npx sass` pays ~**1079 ms**
  (≈ **770× more**) — almost all of that extra second is the `npx` package-resolution
  tax, not dart-sass itself. For a tool that shells out per file, this dominates.
- **Pure compile throughput (same `large.scss`, startup removed): grass ≈ 4.9×
  faster than dart-sass** (21.4 ms vs 105.4 ms/file; 46.8 vs 9.5 files/sec).
  Native Rust beats the dart2js build here. Caveat: the dart-sass 105 ms/file is
  measured with dart-sass's *parallel* multi-isolate batch (user time ≈ 1.5–3.3×
  wall), whereas the grass number is strictly single-threaded sequential — so on
  raw single-core compute the gap is even larger than 4.9×, and a native-AOT
  dart-sass would narrow it somewhat.
- **Cold single large file end-to-end: grass ≈ 13× faster than the dart-sass bin,
  ≈ 59× faster than `npx sass`.**

## Methodology

Two things are deliberately separated:

1. **Process startup cost.** Each `sass <file>` subprocess pays node+dart startup;
   in-process grass pays ~zero. Measured by compiling a ~empty file (`/tmp/tiny.scss`,
   `.a{b:c}`) — that wall time is essentially pure startup. Reported for grass, the
   dart-sass bin, and `npx sass` separately so the npx tax is visible.

2. **Pure compile throughput.** To isolate the parse→eval→serialize cost from
   startup:
   - **grass:** `grass_runner --loop N <file>` recompiles the file N times in one
     in-process loop (re-reading + recompiling each iteration, matching what a
     subprocess does), then divides. N=200 on `large.scss`, N=1000 on the
     hand-written file.
   - **dart-sass:** no in-process mode is available from the CLI, so startup is
     amortized by compiling **K=20 copies of the same `large.scss` in one
     invocation** (`sass dir:dir`) and subtracting one startup: `(wall − 137 ms)/20`.
     Cross-checked against the 40-file batch.

`hyperfine` drove all wall-time measurements (15 runs, 3 warmups; the system was
otherwise quiet). Raw JSON is in `results/*.json`.

### Honesty about noise / variance

- grass startup (~1.4 ms) is below hyperfine's reliable resolution; hyperfine
  warns about this. Treat it as "≈ 1–2 ms, effectively free," not a precise figure.
- `npx sass` startup has high variance (σ ≈ 228 ms, range 0.6–1.6 s) because npx
  does filesystem/package resolution that competes with the OS — outliers were
  flagged. The dart-sass *bin* startup (137 ms ± 3) is the stable, fair number;
  the npx figure is reported only to quantify the wrapper tax.
- dart-sass compute (≥ ms scale) and grass numbers are stable (σ < 3% of mean).
- The dart-sass batch uses multiple isolates (parallelism), so its per-file number
  flatters dart-sass relative to grass's sequential loop; noted above.

## Correctness sanity

Both engines were run on `main.scss` and `large.scss`; outputs were compared after
(a) whitespace normalization (`scripts/normalize_css.sh`) and (b) additionally
canonicalizing color serialization (`scripts/canon_css.py`, folds `rgb()`↔hex,
rounds, lowercases — **without** reordering anything).

Two genuine, well-understood divergences remain. Neither is a correctness bug;
both produce visually identical rendering:

1. **Color serialization format.** dart-sass 1.100 emits CSS Color Level 4-style
   `rgb()` with *fractional* channels (`rgb(159.47…, 23.68…, 20.52…)`); grass 0.13
   emits legacy rounded **hex** (`#9f1815`). Verified these denote the *same* color
   (rounding the dart `rgb()` reproduces the grass hex exactly). This is the bulk of
   the raw diff and is purely cosmetic / spec-version drift.

2. **Declaration hoisting / rule order** (hand-written file). When a rule mixes bare
   declarations with nested rules (here via `@include button-variant` which contains
   `&:hover` etc.), dart-sass **moves the bare declarations into a trailing
   `.btn-primary { … }` block placed after the nested rules** (modern Sass behavior);
   grass 0.13 keeps them inline at the top. Same declarations, same cascade outcome,
   different block layout.

3. **±1 rounding in color math** (generated file). After folding away (1), exactly
   **50 of 25,200 lines** still differ, all colors, all by **≤ 1 unit in a single
   channel** (e.g. `#20285e` vs `#20285f`) — rounding-direction differences in
   `darken`/`lighten` intermediates. **Zero** structural/ordering differences in the
   generated file.

**Verdict:** grass and dart-sass are functionally equivalent on this corpus. The
only differences are (i) `rgb()`-vs-hex serialization, (ii) declaration-hoisting
order, and (iii) ≤1/255 color rounding — all cosmetic.

## Interpretation

- If you invoke a compiler **per file as a subprocess**, startup dominates and the
  in-process Rust engine wins enormously — ~98× vs the dart-sass bin and ~770× vs
  `npx sass`. This is the headline for any build-tool/IDE/save-on-watch scenario.
- On **pure compile**, native Rust (grass) is the faster engine: ~**4.9×** the
  throughput of the dart2js dart-sass build on the same large file, and more than
  that per-core (dart-sass's batch number borrows parallelism). A native-AOT
  dart-sass would close part of the gap but the brief's machine ships the dart2js
  build, where grass clearly wins on raw compile.
